**What is a REST API?**

A **REST API** (Representational State Transfer Application Programming Interface) is a way for different software applications to communicate with each other over a network, typically the internet. It's built upon the principles of **REST**, which is an architectural style, not a specific protocol or standard. REST defines a set of constraints for designing networked applications, primarily focused on how resources are accessed and manipulated.

**Core Principles of REST:**

1.  **Client-Server:** A clear separation exists between the client (requesting data or actions) and the server (providing data or performing actions). They can evolve independently.
2.  **Statelessness:** Each request from a client to the server must contain all the information necessary for the server to understand and fulfill the request. The server does not store any client context between requests. This improves scalability and reliability.
3.  **Cacheability:** Responses from the server should indicate whether they can be cached by clients or intermediaries. This improves performance by allowing clients to reuse previously fetched data.
4.  **Uniform Interface:** This is a key constraint that simplifies and decouples the architecture. It has several sub-constraints:
    *   **Resource Identification:** Resources (e.g., a user profile, a list of tasks) are identified by URIs (Uniform Resource Identifiers).
    *   **Resource Manipulation Through Representations:** Clients interact with resources by exchanging representations of those resources (commonly in JSON or XML format).
    *   **Self-Descriptive Messages:** Each message contains enough information to understand how to process it, including the method (GET, POST, PUT, DELETE) and the media type.
    *   **HATEOAS (Hypermedia as the Engine of Application State):** Resources should include links to related actions or resources, guiding the client through the application's state. (Often partially implemented).
5.  **Layered System:** A client cannot tell whether it's connected directly to the end server or to an intermediary (like a load balancer or proxy). This allows for flexibility in system architecture.

**How a REST API Works in Practice:**

A REST API uses standard HTTP methods to perform actions on resources identified by URIs.

*   **Resources:** Everything is a resource (e.g., a specific user ` /users/123 ` or a collection of tasks ` /tasks `).
*   **HTTP Methods (Verbs):** These are used to define the action to be performed on a resource:
    *   `GET`: Retrieve a resource or collection.
    *   `POST`: Create a new resource.
    *   `PUT`: Update/replace a resource entirely.
    *   `PATCH`: Partially update a resource.
    *   `DELETE`: Remove a resource.
*   **Representations:** Data is exchanged in a representation format, most commonly **JSON**.
*   **Stateless Communication:** The server treats each request independently. If state is needed (like user authentication), it's sent with every request (e.g., via tokens in headers).
*   **Status Codes:** Servers respond with HTTP status codes (e.g., `200 OK`, `201 Created`, `404 Not Found`, `500 Internal Server Error`) to indicate the result of the request.

**Example:**

To get a list of all tasks, a client might send a `GET` request to the URI ` /api/tasks `. The server would respond with a `200 OK` status and a JSON representation of the tasks. To create a new task, a client might send a `POST` request to ` /api/tasks ` with a JSON body describing the new task.
